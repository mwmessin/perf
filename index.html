networking!

Naive serial networking

      resource1 = HTTP.get("www.host.com/resource1")
      resource2 = HTTP.get("www.host.com/resource2")
      resource3 = HTTP.get("www.host.com/resource3")

      render_response(resource1, resource2, resource3)

blocking networking calls

the default, worst case performance, many ways to do better

Request combination

the fastest request is no request!

bundle queries/data together, return it all at once

avoid HTTP overhead; persistent HTTP connections

Parallelization

      resources = [
        "www.host.com/resource1",
        "www.host.com/resource2",
        "www.host.com/resource3",
      ]

      results = resources.in_threads do |url|
        HTTP.get(url)
      end

      render_response(*results)

MRI Ruby has the GIL

threads are meant for blocking code

Resource dependencies

can't parallelize all requests

      resourcesA = [
        "www.host.com/resource1",
        "www.host.com/resource2",
        "www.host.com/resource3",
      ]

      resultsA = resourcesA.in_threads do |url|
        HTTP.get(url)
      end

      resourcesB = [
        "www.host.com/resource4",
        "www.host.com/resource5",
      ]

      resultsB = resourcesB.in_threads do |url|
        HTTP.post(url, { data: resultsA[0] })
      end

      render_response(*(resultsA + resultsB))

end up with sets of requests which can be done in threads

ordering requests to maintain dependencies is a code maintenance pain point

Groups of parallelized requests

long pole in each group can introduce time waste

have to shuffle requests into different sets to optimize

Build a DAG of dependencies

a node for each resource

parallelize all nodes which have no unresolved dependencies

resolve a dependency for all dependent nodes upon resource-request completion

...repeat until graph is fully resolved

Dependency graphs

execute 1,2,3

5 will run immediately once 1 and 3 finish (doesn't care about order)

Fully optimized timing diagrams

as good as it gets

Applies to more than just networking

      resolve(
        1 => [[], ->{ resource1 = HTTP.get(url1) }],
        2 => [[], ->{ resource2 = HTTP.get(url2) }],
        3 => [[], ->{ resource3 = HTTP.get(url3) }],
        4 => [[1], ->{ resource4 = HTTP.post(url4, { data: resource1 }) }],
        5 => [[1,3], ->{ resource5 = HTTP.post(url5, { data: [resource1, resource3] }) }],
      )

      render_response(resource1, resource2, resource3, resource4, resource5)

build the DAG off of Procs, wrap any call in a lambda

any blocking call: DB, Filesystem, ... even a mix
